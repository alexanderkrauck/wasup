## Systemic findings (patterns across scenarios)

1. **The events/offerings boundary is the biggest coverage hole** (S1, S3, S9, S19). Real users don't distinguish "event" from "thing I can do": a museum being open, a sauna, a drop-in climbing session. Fix: `event.kind = one_off | series | course | festival | standing_offering` - standing offerings have opening-hour-style rrules and no scarcity. Cheap to add now, brutal to retrofit.
2. **Logistics attributes are missing** (S2, S6, S9, S17, S20): `registration_required`, `registration_deadline`, `booking_url`, `drop_in_ok`, `doors_at`, `late_entry_ok`, `participation_mode`. These decide whether a person can *act* on a listing - half the scenarios stumbled here.
3. **Hierarchy**: `parent_event_id` for festivals/sub-events (S8).
4. **Privacy/takedown is a real gap with legal teeth** (S18): needs suppression heuristics (residential address + small venue + personal-profile organizer → don't publish geo/address), a takedown endpoint with SLA, and GDPR-aware storage of anything person-identifying.
5. **Rights metadata** (S11): mark scraped descriptions/images as `rights=quoted` vs `rights=licensed`; API consumers that republish need to know. Summaries generated by us are ours; scraped prose is not.
6. **Honest unknowns as API contract** (S20, S15, S13): every nullable field's null means "unknown", never "no"; the doc should state this and consumers must be able to filter `where ends_at is not null`.
7. **Staleness decay** (S14): canon confidence should decay when a source misses its expected re-confirmation cadence, not only on explicit failure.
8. **Negative constraints need guarantees** (S22): category/tag exclusions must be set-logic filters, with embeddings only ranking within the allowed set - never the other way around.

## What survived contact intact

Cancellation propagation (S7), rumor/provenance handling (S21), zombie-listing hygiene (S14), the loneliness/newcomer use case (S16), long-tail + multilingual coverage (S4), and inverse-demand signaling via the zero-result monitor (S19). Notably these are the doc's most distinctive parts - the claims-vs-canon + confidence + QA-loop core holds up under adversarial cases.
